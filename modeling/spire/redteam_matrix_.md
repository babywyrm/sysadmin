
# Companion Matrix: Attack Scenarios vs SPIFFE/SPIRE Mitigations ..(RC1)..

## Purpose

This matrix maps realistic internal attack scenarios to:

- attacker objectives
- trust failures being exploited
- where SPIFFE/SPIRE materially help
- where SPIFFE/SPIRE are not sufficient on their own
- what additional controls are still required

The goal is to clarify a common misunderstanding:

> SPIFFE/SPIRE greatly strengthen workload identity, but they are not a complete substitute for authorization, policy, or application safety controls.

---

## Scenario Matrix

| Attack scenario | Attacker goal | Likely failure mode exploited | How SPIFFE/SPIRE help | Remaining gap | Recommended compensating controls |
|---|---|---|---|---|---|
| Forged internal service headers | Pretend to be a trusted internal service | Target trusts caller-supplied headers such as `X-Service`, `X-Role`, or `X-SPIFFE-ID` | Verified workload identity allows the receiver to ignore forged claims and trust cryptographic identity instead | If the app still trusts raw headers, spoofing remains possible | Never trust raw identity headers from callers; verify peer identity via mTLS or trusted proxy context |
| Service impersonation | Act as a privileged workload such as an orchestrator or control API | Identity is inferred from naming, IP, or weak metadata rather than verified credentials | Makes successful impersonation much harder by requiring valid signed identity material | If private keys or signing chain are compromised, impersonation may still occur | Protect SPIRE infrastructure, secure private keys, use short-lived credentials, audit identity issuance |
| Lateral movement inside cluster | Reach internal APIs after initial compromise | Internal traffic is implicitly trusted because it originates from inside the cluster or namespace | Shifts trust from network location to verified workload identity | A real but overprivileged workload can still move where policy allows | Fine-grained authorization, service segmentation, least privilege per workload |
| Namespace trust abuse | Use shared namespace membership to access services | Services trust namespace or service account grouping too broadly | Provides more specific workload identity than namespace membership alone | If authorization remains coarse, broad trust still exists | Per-service authorization rules, distinct service accounts, tighter identity granularity |
| IP-based allowlist abuse | Leverage approved network position | Network location treated as identity | Replaces location-based trust with cryptographic identity | Legacy systems may still rely on IP allowlists | Prefer identity-based authz over network trust; restrict fallback paths |
| Fake approval or policy context | Claim that an action was pre-approved | Receiver trusts unsigned approval metadata in headers or payloads | Lets receiver bind trust to actual caller identity rather than caller narrative | SPIFFE proves who called, not whether approval claims are true | Signed approval tokens, server-side policy checks, workflow validation |
| AI agent hallucinates authority | Claim elevated role or system status it does not have | Receiver trusts role text or context generated by an autonomous system | Allows receiving service to verify actual workload identity independently of generated claims | A legitimate workload may still be allowed to overstate intent or context | Identity-based authorization, strict role mapping, distrust model-generated authority assertions |
| Prompt-injected tool use | Trick a legitimate agent into calling a sensitive tool | Tool execution is overly trusted once the caller is authenticated | Helps identify exactly which workload invoked the tool | If the tool is allowed for that workload, misuse can still occur | Tool-level authorization, scoped permissions, approval gates for sensitive operations |
| Legitimate workload misuse | Use valid credentials to perform harmful actions | Authorization is too broad or action-level controls are weak | Improves attribution and makes misuse auditable | Does not block harmful but authorized behavior | Least privilege, action-level policy, confirmations, break-glass controls |
| Overprivileged orchestrator | Abuse broad control-plane permissions | Powerful workload has too many rights across the platform | Ensures requests are attributable to the actual orchestrator | If orchestrator is compromised or misbehaves, damage may still be large | Permission reduction, scoped service identities, approval workflows for destructive actions |
| Replay of believable internal identity claims | Reuse old names, metadata, or patterns to gain trust | Humans or apps trust familiar internal labels more than verified identity | Encourages trust in signed workload identity instead of names or labels | Social and operational trust shortcuts may still bypass verification | Standardize identity verification in services and proxies; train operators not to trust labels alone |
| Direct access to internal admin API | Reach privileged API from a non-privileged workload | Admin API only checks network reachability or partial trust indicators | Requires caller to present valid workload identity before access | If admin API authorization is too broad, verified callers may still do too much | Identity-based API authorization, route restrictions, admin-only policy enforcement |
| Fake SPIFFE header injection | Set `X-SPIFFE-ID` or similar manually | Receiver confuses a plain header with verified identity | Real SPIFFE verification exposes the difference between claimed and actual identity | Unsafe designs may still trust the header | Only accept identity from mTLS, signed JWT-SVIDs, or trusted mesh/proxy metadata |
| Compromised workload using real SVID | Use its own valid identity to attack downstream systems | Workload has legitimate identity and excessive permissions | Makes incident attribution and policy reasoning better | Does not stop a valid identity from abusing allowed actions | Short TTLs, strong authz, anomaly detection, workload isolation |
| Stolen credential or key material | Reuse valid identity credentials to impersonate a service | Private key compromise or secret exposure | Short-lived credentials reduce exposure window; trust model is stronger than static secrets | Stolen valid material can still be abused until expiry | Protect keys, rotate aggressively, isolate workloads, monitor anomalous identity use |
| Compromise of SPIRE control plane | Issue fraudulent identities or abuse trust root | Trust authority itself is compromised | No identity system can fully protect against compromise of its own signing authority | Blast radius becomes significant | Harden SPIRE server, restrict admin access, audit issuance, use defense in depth |
| Broken verifier implementation | Skip cert validation or trust the wrong signal | App incorrectly verifies identity or trusts unverified proxy data | SPIFFE/SPIRE only help if verification is done correctly | Misconfiguration or bad code can nullify benefits | Use well-tested libraries, centralize verification, validate trust chains consistently |
| Bypass around mesh or identity layer | Reach service over an unprotected path | Some network path avoids mTLS or proxy verification | Strong identity on protected paths still helps | Unprotected side paths remain exploitable | Enforce mesh participation, block direct paths, network policy, service exposure review |

---

## Quick interpretation guide

Use the matrix this way:

- If the problem is impersonation, spoofing, or weak internal trust:
  - SPIFFE/SPIRE are often a strong mitigation

- If the problem is overbroad permissions or harmful use of real permissions:
  - SPIFFE/SPIRE help with attribution, but they are not enough

- If the verifier is weak or bypassed:
  - even strong identity systems can fail operationally

---

## Practical summary

### Scenarios where SPIFFE/SPIRE are especially valuable

SPIFFE/SPIRE are high-value controls when the attacker is trying to:

- pretend to be a different workload
- exploit trust in internal headers
- move laterally using “inside the cluster” trust
- abuse weak service identity assumptions
- fool downstream systems with plausible service claims

### Scenarios where more is needed

SPIFFE/SPIRE are not sufficient alone when the attacker is:

- using a real workload identity with valid permissions
- exploiting prompt injection or unsafe tool design
- abusing a control-plane workload with broad privileges
- benefiting from weak authorization or weak business logic

---

## Short conclusion

The companion matrix shows the central tradeoff clearly:

> SPIFFE/SPIRE are excellent at strengthening trust in who a workload is.

But:

> The safety of what that workload is allowed to do still depends on authorization, policy, and application design.
